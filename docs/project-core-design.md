# Vibeuddy 项目核心设计文档

## 1. 项目定位

Vibeuddy 不是普通桌宠，也不是普通 AI 编程工具，而是一个面向 Vibe Coding 的项目生命体反馈系统。

在 Vibe Coding 中，用户会不断输入需求、修改想法、生成功能。传统工具通常只反馈代码、日志和报错，但很多用户并不能直观看到：

- 项目现在是否清晰？
- 项目是否变得更好玩？
- 项目是否越来越复杂？
- 项目是否存在稳定风险？
- 项目是否值得继续推进？

Vibeuddy 用一只 AI 项目宠物承载这些抽象反馈。每一次用户按下回车，Agent 都会评估当前项目状态，并把项目变化转化为宠物的形态、情绪、评分和成长变化。

核心表达：

> 每一次回车，项目都会长成一只更具体的宠物。

## 2. 核心体验

Vibeuddy 想让用户看到的不是冷冰冰的项目分析，而是一个项目被逐步孵化和养成的过程。

完整体验链路：

```text
输入想法 -> 按下回车 -> Agent 评估项目 -> 宠物状态变化 -> 显示评分 -> 生成成长记录 -> 输出项目宠物海报
```

用户感受到的不是“我在写项目文档”，而是：

> 我正在孵化一个项目生命体。

这个体验同时提供两类价值：

- 好玩：宠物会开心、迷茫、受伤、过载、进化，形成养成感。
- 好用：宠物背后的状态变化来自 Agent 对项目清晰度、完成度、趣味性、稳定性和传播性的评估。

因此，Vibeuddy 的核心不是“宠物可爱”，而是：

> 宠物是项目状态的可视化表达。

## 3. 系统顶层架构

系统分为四个核心模块。

### 3.1 输入层：Vibe Coding 交互区

用户通过输入框持续输入需求或修改想法，例如：

- 我想做一个线下陌生人社交游戏。
- 加一个扫码互动功能。
- 增加排行榜和分享海报。
- 帮我简化 MVP。

这一层模拟 Vibe Coding 中“回车触发项目迭代”的体验。

核心交互文案：

```text
Press Enter to evolve your project pet.
```

### 3.2 Agent 评估层：项目状态判断

用户按下回车后，Agent 分析本轮输入对项目的影响，并从 5 个维度进行评估：

| 维度 | 含义 |
| --- | --- |
| Clarity 清晰度 | 项目是否更容易被理解 |
| Completion 完成度 | 项目是否更接近可展示 Demo |
| Fun 趣味性 | 用户是否更想继续玩 |
| Stability 稳定性 | 方案是否更容易实现、风险是否更少 |
| Shareability 传播性 | 其他人看到后是否愿意分享或邀请朋友体验 |

Agent 输出内容包括：

- 当前宠物状态
- 五维评分
- 本轮分数变化
- 本轮变化总结
- 项目风险提示
- 下一步建议
- 是否触发进化

### 3.3 宠物表现层：UI 与动画核心

前端根据 Agent 返回的 `petState` 切换宠物形象和动画。

宠物状态设计：

| 状态 | 含义 | 视觉表现 |
| --- | --- | --- |
| egg | 初始蛋，项目还只是一个想法 | 发光、轻微晃动、裂纹 |
| idle | 正常待机，项目稳定成长 | 上下漂浮、核心微光 |
| happy | 本轮输入让项目更清晰或更有趣 | 跳动、亮眼、小星星 |
| confused | 需求模糊，方向不明确 | 歪头、问号、左右晃动 |
| injured | 稳定性下降，出现实现风险 | 变暗、核心闪烁、裂纹 |
| overload | 功能堆叠，MVP 变重 | glitch、错位、紧张表情 |
| evolve | 达成关键突破，进入下一阶段 | 光圈、粒子、白光闪烁 |
| final | 形成清晰方案，可以生成海报 | 完全体展示、稳定发光 |

UI/动画负责人的核心工作是把这些状态做成用户一眼能理解的视觉语言。

### 3.4 结果层：评分与海报

当用户完成多轮迭代后，系统生成最终结果：

- 一只最终项目宠物
- 一个项目总评分
- 一张可分享海报
- 一段项目总结

示例结果：

```text
你的项目宠物：赛博灵感猫
当前阶段：Lv.4 成长期
项目评分：82
最强属性：趣味性 91
当前弱点：稳定性 64
宠物评语：这个项目很有传播感，但需要进一步收敛 MVP。
```

这让项目从过程反馈变成可展示、可传播的成果。

## 4. UI 页面设计

### 4.1 页面一：项目蛋首页

目标：建立第一印象，让用户知道这是一个会被输入孵化的项目生命体。

核心元素：

- 居中的发光项目蛋
- 输入框
- 初始引导语

页面文案：

```text
Press Enter. Hatch your project.
Tell me your project idea...
```

交互：

- 用户输入第一个想法并按下回车
- 项目蛋发光
- 蛋体震动
- 裂纹出现
- 进入宠物主界面

视觉氛围：

- 白色或浅灰背景
- 轻赛博蓝光效
- 像素风，但避免廉价感
- 可爱、治愈、轻科技

### 4.2 页面二：宠物主界面

目标：承载主要使用流程，是产品的核心页面。

布局：

```text
┌────────────────────────────────────────┐
│              Project Pet               │
│                                        │
│   左侧评分       中间宠物       右侧日志 │
│                                        │
│           输入框 Press Enter            │
└────────────────────────────────────────┘
```

区域说明：

- 中间：宠物主体，需要最大、最醒目，是产品记忆点。
- 左侧：五维项目属性评分。
- 右侧：Agent 评估日志。
- 底部：输入框，承载持续 Vibe Coding。

左侧评分示例：

```text
清晰度 72
完成度 58
趣味性 81
稳定性 64
传播性 77
```

右侧日志示例：

```text
本轮变化：项目增加了传播机制。
Agent 判断：趣味性提升，但复杂度略升。
下一步建议：先做扫码闭环，不要急着做排行榜。
```

### 4.3 页面三：进化瞬间页或弹窗

目标：制造现场展示的记忆点。

触发条件示例：

- Lv.1 项目蛋 -> Lv.2 初始宠物
- Lv.2 初始宠物 -> Lv.3 成长期宠物
- Lv.3 成长期宠物 -> Lv.4 完全体

动画流程：

```text
屏幕微暗 -> 宠物发光 -> 像素粒子散开 -> 白光闪烁 -> 新形态出现 -> 等级提升
```

提示文案：

```text
Your project has evolved.
你的项目进化了。
```

### 4.4 页面四：负面状态与修复反馈

目标：当项目风险上升时，用可爱但明确的方式提示用户修复。

状态示例：

- 迷茫：眼睛变成问号，头轻微左右晃动。
- 受伤：身体变暗，心形核心闪烁。
- 过载：周围出现 glitch 像素闪烁。

注意：负面状态不能恐怖，仍然要保持可爱和可修复感。

文案示例：

```text
它有点过载了。你的项目功能变多了，但 MVP 变得不清晰。
```

修复建议示例：

```text
试试输入：“帮我压缩成 3 个核心功能。”
```

这一页的重点是让用户产生“我可以救回这个项目宠物”的行动欲望。

### 4.5 页面五：最终海报页

目标：生成可展示、可分享的项目宠物报告卡。

海报内容：

- 项目宠物形象
- 项目名称
- 总评分
- 五维雷达图或进度条
- 进化次数
- 最强属性
- 当前弱点
- 宠物评语

示例：

```text
VibePet Report

我的项目宠物：Cyber Hatch Cat
项目评分：82
最强属性：趣味性
当前弱点：稳定性
进化次数：5
本轮评语：这是一个很有传播潜力的项目，但需要收敛技术范围。
```

最终海报是产品的传播点。

## 5. 动画顶层设计

动画采用状态驱动，不做复杂骨骼动画。每种宠物状态绑定一种简单但明确的 CSS 动效。

### 5.1 Idle Float 待机漂浮

用途：平时待机。

表现：

- 宠物上下轻轻漂浮
- 心形核心微弱闪烁
- 尾巴轻微摆动或两三帧切换

意义：

```text
项目处于正常生长状态。
```

### 5.2 Enter Pulse 输入反馈

用途：用户按下回车。

表现：

- 输入框发光
- 能量流向宠物
- 宠物轻微放大再恢复
- 身上青蓝色光点闪一下

意义：

```text
这次 prompt 被宠物吸收了。
```

### 5.3 Happy Bounce 开心跳动

用途：项目变清晰、变好玩。

表现：

- 宠物跳一下
- 眼睛变亮
- 周围出现小星星或像素粒子
- 评分条上涨

意义：

```text
本次修改是正向迭代。
```

### 5.4 Confused Shake 迷茫晃动

用途：需求不清晰。

表现：

- 宠物头歪一下
- 身体轻轻左右晃动
- 头上出现问号像素
- 清晰度下降或保持不变

意义：

```text
用户需要更明确地描述项目。
```

### 5.5 Injured Flicker 受伤闪烁

用途：稳定性下降。

表现：

- 宠物整体变暗
- 心形核心闪烁
- 出现小裂纹或低电量 icon
- 稳定性条下降

意义：

```text
项目可实现性或技术稳定度存在风险。
```

### 5.6 Overload Glitch 过载故障

用途：功能堆太多。

表现：

- 宠物周围出现 glitch 像素
- 身体短暂错位
- 眼睛变成螺旋或紧张表情
- 出现复杂度提示

意义：

```text
项目变得太重，需要收敛。
```

### 5.7 Evolve Flash 进化闪光

用途：达到关键节点。

表现：

- 宠物被光圈包围
- 像素粒子向外扩散
- 白光一闪
- 切换到新形态
- 出现等级提升

意义：

```text
项目完成一次阶段性成长。
```

## 6. Agent 与 UI 对接协议

前后端通过固定 JSON 解耦。Agent 负责判断项目状态，UI 负责状态视觉化。

示例返回：

```json
{
  "petState": "happy",
  "evolutionStage": 2,
  "score": 78,
  "metrics": {
    "clarity": 82,
    "completion": 61,
    "fun": 88,
    "stability": 66,
    "shareability": 79
  },
  "delta": {
    "clarity": 8,
    "completion": 5,
    "fun": 12,
    "stability": -3,
    "shareability": 7
  },
  "summary": "这个改动让项目更有趣，也更适合现场传播。",
  "risk": "功能数量增加后，MVP 可能变得偏重。",
  "suggestion": "建议先完成扫码互动和宠物反馈闭环。"
}
```

状态映射：

| petState | UI 表现 |
| --- | --- |
| egg | 项目蛋 |
| idle | 待机漂浮 |
| happy | 开心跳动 |
| confused | 迷茫晃动 |
| injured | 受伤闪烁 |
| overload | glitch 过载 |
| evolve | 进化光效 |
| final | 完全体展示 |

解耦原则：

- Agent 只负责状态判断。
- UI 只负责状态视觉化。
- 动画只依赖 `petState`、`evolutionStage`、`metrics` 和 `delta`。

## 7. UI/动画负责人任务清单

### 7.1 定视觉风格

推荐方向：

```text
8-bit 像素风 + 可爱治愈 + 轻赛博蓝
```

设计关键词：

- 轻科技
- 可爱
- 治愈
- 成长感
- 像素粒子
- 发光核心

### 7.2 定宠物状态图

至少准备以下状态：

- egg
- idle
- happy
- confused
- injured
- overload
- evolve 或 final

### 7.3 定动画规则

每个状态绑定一个 CSS 动效，优先保证状态清晰，不追求复杂动画。

推荐实现方式：

- CSS keyframes
- 状态 class 切换
- 少量粒子元素
- 评分条 transition
- 进化弹窗 overlay

### 7.4 定主界面布局

主界面必须保证：

- 宠物在视觉中心
- 左侧评分一眼可读
- 右侧日志能说明本轮变化
- 输入框始终可见
- 回车反馈足够明显

### 7.5 定最终海报

海报应当成为传播物料，包含：

- 宠物形象
- 项目名
- 总评分
- 五维指标
- 进化次数
- 最强属性
- 当前弱点
- 宠物评语

## 8. MVP 闭环

最小可展示版本需要完成以下闭环：

```text
用户输入 -> 回车 -> 模拟 Agent 返回 JSON -> 宠物吸收能量 -> 状态变化 -> 分数变化 -> 日志反馈 -> 触发进化或负面修复建议
```

MVP 不需要真实复杂 Agent，也可以先用规则模拟：

- 输入包含“简化”“MVP”“核心功能”：提升清晰度和稳定性。
- 输入包含“分享”“海报”“邀请”：提升传播性和趣味性。
- 输入包含“排行榜”“AI”“地图”“实时”“多人”：提升趣味性，但可能降低稳定性。
- 输入过短或模糊：触发 confused。
- 功能连续增加过多：触发 overload。

只要这个闭环成立，项目概念就成立。

## 9. 答辩表达

完整表达：

> 我们做的是一个 Vibe Coding 项目生命体系统。用户每一次输入需求并按下回车，Agent 都会评估项目的清晰度、完成度、趣味性、稳定性和传播性，并把评估结果转化为宠物的形态、情绪、动画和评分。它让抽象的项目成长过程变得可视化、游戏化、可感知，使用户像养宠物一样理解自己的项目正在如何进化。

简短表达：

> 我们把 Vibe Coding 的项目迭代过程变成了一只会成长、会生病、会进化的项目宠物。它既让创作过程更好玩，也让用户更直观地理解项目状态。

UI/动画展示重点：

```text
回车 -> 宠物吸收能量 -> 状态变化 -> 评分反馈 -> 进化或受伤 -> 修复建议
```

只要这个闭环在现场被看见，Vibeuddy 的核心价值就能被理解。
