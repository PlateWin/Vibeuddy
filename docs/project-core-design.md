# Vibeuddy 项目核心设计文档

## 1. 项目定位

Vibeuddy 不是普通桌宠，也不是普通 AI 编程工具，而是一只**会陪你 Vibe Coding 的 AI 项目桌宠**。

Vibe Coding 让编程变得像对话一样简单，但它也带来了两个深层问题：

- **空心化**：AI 帮你快速生成代码，用户越来越像在"点菜"——加登录、加排行榜、加聊天、加分享、加数据库……但可能从未真正思考过"为什么加？服务谁？是否必要？会不会拖慢 MVP？"
- **执行者陷阱**：用户沉迷于"让 AI 做出来"，却逐渐失去对项目的理解权和决策权，从"创作者"滑落为"点菜员"。

传统工具只反馈代码、日志和报错。它们告诉你能不能跑，却不告诉你该不该做。

Vibeuddy 用一只 AI 项目宠物同时解决这两个问题。它有三个核心能力：

1. **项目状态反馈**——每次回车后评估项目的清晰度、完成度、趣味性、稳定性、传播性，并转化为宠物的形态、情绪和动画变化。
2. **项目理解追问**——不像工具一样沉默执行，而是主动向用户提问，帮助用户思考项目目标、功能取舍、用户体验和风险。"这个功能解决的是真实痛点，还是只是看起来很酷？"
3. **反空心化陪伴**——在 AI 编程过程中充当"思考提醒器"，在关键节点追问用户，让用户保留对项目的理解权和决策权。

更准确地说，宠物扮演的角色是：

> **它是 AI 编程过程中的思考教练。**

Vibeuddy 把沉默的代码变成会说话的老师，把养成感变成学习驱动力，把"边写边学"变成实时介入。宠物不是为了可爱而存在——养成感不是装饰，而是思考的外在驱动力。

核心表达：

> 每一次回车，项目都会长成一只更具体的宠物。每一次追问，你都会成为更清醒的创作者。

## 2. 核心体验

Vibeuddy 想让用户获得的不只是”看到项目状态”的反馈，而是一套**边写边学的实时介入体验**。

```text
输入想法 → 按下回车 → Agent 评估项目 → 宠物状态变化 → 宠物主动追问 → 用户思考并回应 → 项目迭代 → 生成成长记录 → 输出项目宠物海报
```

用户感受到的不是”我在写项目文档”，也不是”AI 在帮我写代码”，而是：

> 我正在孵化一个项目生命体，而它也在陪我思考。

这个体验同时提供三类价值：

- **好玩**：宠物会开心、迷茫、受伤、过载、进化，形成养成感。养成感不是装饰，而是让用户愿意持续面对项目问题的情感驱动力。
- **好用**：宠物背后的状态变化来自 Agent 对项目清晰度、完成度、趣味性、稳定性和传播性的评估，让抽象问题变得可视、可感、可修。
- **好清醒**：宠物不会沉默执行所有指令。它会在用户盲目堆功能时追问”这个功能解决真实痛点，还是只是看起来很酷？”，把用户从执行者拉回创作者。

因此，Vibeuddy 的核心不是”宠物可爱”，也不是”Agent 能打分”，而是：

> 宠物是项目状态的可视化表达，也是项目思考的主动触发者。

## 3. 三大核心能力

Vibeuddy 宠物的行为不是随机的，而是由三个递进的核心能力驱动。

### 3.1 能力一：项目状态反馈

用户每次按下回车后，Agent 分析本轮输入对项目的影响，并从 5 个维度更新项目状态：

| 维度 | 含义 | 宠物表现 |
| --- | --- | --- |
| 清晰度 Clarity | 项目是否更容易被理解 | 上升→开心；下降→迷茫 |
| 完成度 Completion | 项目是否更接近可展示 Demo | 上升→进化接近 |
| 趣味性 Fun | 用户是否更想继续玩 | 上升→开心跳动 |
| 稳定性 Stability | 方案是否更容易实现、风险是否更少 | 下降→受伤闪烁 |
| 传播性 Shareability | 其他人看到后是否愿意分享 | 上升→核心发光 |

宠物根据本轮评估结果切换到对应情绪状态（happy / confused / injured / overload / evolve / final）。

这是 Vibeuddy 的基础层——让用户**看见**项目正在发生什么。

### 3.2 能力二：项目理解追问

AI 不只是评价项目，还会围绕项目**主动提问**，防止用户在 Vibe Coding 里只顾着让 AI 加功能。

追问在以下时机触发：

- **连续三轮加分功能**：宠物判断用户可能在盲目堆功能，触发追问。
- **稳定性持续下降**：触发"这个方向还能做 MVP 吗？"类追问。
- **方向跳跃**：用户本轮输入与上一轮方向明显不同，触发"你是在 pivot 还是在分心？"类追问。
- **长时间未输入核心定义**：触发"你还没说过这个项目是给谁用的"类追问。

示例追问库：

```text
项目定位类：
  "这个功能解决的是用户的真实痛点，还是只是看起来很酷？"
  "如果明天就要演示，你现在最应该完成哪一个闭环？"
  "用户第一次打开这个项目，三秒内能理解它是干什么的吗？"
  "你的核心用户是谁？他们为什么会愿意继续使用？"

功能取舍类：
  "这个功能会让项目更好，还是只是让项目更复杂？"
  "如果删掉这个功能，项目还成立吗？"
  "有没有比这个更简单的方式达到同样的效果？"

风险认知类：
  "这个项目最容易失败的地方在哪里？"
  "如果只剩一天开发时间，你会优先保留哪些功能？"
  "现在的技术方案里，最不确定的部分是什么？"

体验反思类：
  "你自己作为用户，会愿意把这个项目推荐给朋友吗？为什么？"
  "这个项目的'啊哈时刻'是什么？用户在哪个瞬间会觉得'这个有意思'？"
```

追问的作用：**把用户从"执行者"拉回"创作者"**。

### 3.3 能力三：反空心化陪伴

Vibe Coding 的最大问题之一是：AI 可以帮你快速生成代码，但也可能让用户失去主动判断。

用户会越来越像在"点菜"：

```text
加登录。加排行榜。加聊天。加分享。加数据库。加 AI。加实时。加地图。
```

但他可能没有真正思考过每道"菜"是否必要。

宠物承担的角色是：**AI 编程过程中的思考提醒器**。

反空心化的具体机制：

- **连续加功能检测**：当用户连续 N 轮只提新增功能而不做收敛，宠物主动介入。
- **"为什么"追问**：在关键决策节点（新增模块、引入外部依赖、改变核心交互），宠物会问"为什么加这个"。
- **MVP 边界提醒**：当功能数量超过 MVP 合理范围，宠物进入 overload 状态，同时追问"哪些功能可以留给 V2？"
- **决策权返还**：宠物不会替用户做决定，而是通过提问让用户自己做出判断，保留用户对项目的理解权和决策权。

```text
宠物不会说："不要加排行榜。"
宠物会说："排行榜服务的是谁？他们为什么需要看排名？如果你的核心用户只是两个人面对面扫码，排行榜对他们重要吗？"
```

**三个能力的关系**：

```text
能力一（反馈）→ 让用户看见项目状态
       ↓
能力二（追问）→ 让用户思考项目方向
       ↓
能力三（陪伴）→ 让用户保留项目决策权
```

三个能力叠在一起，宠物从"项目仪表盘"变成了"项目思考教练"。

## 4. 系统顶层架构

系统分为四个核心模块。

### 4.1 输入层：Vibe Coding 交互区

用户通过输入框持续输入需求或修改想法，例如：

- 我想做一个线下陌生人社交游戏。
- 加一个扫码互动功能。
- 增加排行榜和分享海报。
- 帮我简化 MVP。

这一层模拟 Vibe Coding 中“回车触发项目迭代”的体验。

核心交互文案：

```text
Press Enter to evolve your project pet.
```

### 4.2 Agent 评估层：项目状态判断 + 理解追问

Agent 在本层完成两件事：**状态评估** 和 **主动追问**。

**评估维度**

| 维度 | 含义 |
| --- | --- |
| Clarity 清晰度 | 项目是否更容易被理解 |
| Completion 完成度 | 项目是否更接近可展示 Demo |
| Fun 趣味性 | 用户是否更想继续玩 |
| Stability 稳定性 | 方案是否更容易实现、风险是否更少 |
| Shareability 传播性 | 其他人看到后是否愿意分享或邀请朋友体验 |

**追问触发规则**

| 触发条件 | 追问类型 | 示例问题 |
| --- | --- | --- |
| 连续 3 轮加分功能且 stability 下降 | 功能取舍 | "这些功能里，哪个是明天演示必须有的？" |
| 用户方向跳跃（本轮与上轮主题显著不同） | 定位确认 | "你是在 pivot 方向，还是只是想到一个新点子？" |
| 连续 5 轮未定义目标用户 | 用户认知 | "目前为止你还没描述过你的核心用户是谁。" |
| 功能数 > MVP 合理阈值 | 范围收敛 | "如果只剩 3 个功能，你保留哪 3 个？" |
| 引入外部依赖或复杂技术 | 风险认知 | "这个方案里最不确定的部分是什么？" |
| 长时间未触发正向状态变化 | 项目反思 | "这个项目最容易失败的地方在哪里？" |

Agent 输出内容包括：

- 当前宠物状态
- 五维评分
- 本轮分数变化
- 本轮变化总结
- 项目风险提示
- 下一步建议
- 是否触发进化
- **本轮追问（可能为空）**
- **追问触发原因**

### 4.3 宠物表现层：UI 与动画核心

前端根据 Agent 返回的 `petState` 切换宠物形象和动画。

宠物状态设计：

| 状态 | 含义 | 视觉表现 |
| --- | --- | --- |
| egg | 初始蛋，项目还只是一个想法 | 发光、轻微晃动、裂纹 |
| idle | 正常待机，项目稳定成长 | 上下漂浮、核心微光 |
| happy | 本轮输入让项目更清晰或更有趣 | 跳动、亮眼、小星星 |
| confused | 需求模糊，方向不明确 | 歪头、问号、左右晃动 |
| injured | 稳定性下降，出现实现风险 | 变暗、核心闪烁、裂纹 |
| overload | 功能堆叠，MVP 变重 | glitch、错位、紧张表情 |
| evolve | 达成关键突破，进入下一阶段 | 光圈、粒子、白光闪烁 |
| asking | 宠物向用户追问项目问题 | 歪头、眼睛闪烁问号、气泡弹出问题文字 |
| final | 形成清晰方案，可以生成海报 | 完全体展示、稳定发光 |

`asking` 是新增的关键状态——宠物不是沉默执行，而是主动提问。进入此状态时宠物会：
1. 停止当前动画，切换到歪头+眼睛闪烁问号
2. 气泡弹出追问文字
3. 等待用户回应后切换回对应状态（回答清晰→happy；回答模糊→confused）

UI/动画负责人的核心工作是把这些状态做成用户一眼能理解的视觉语言。

### 4.4 结果层：评分与海报

当用户完成多轮迭代后，系统生成最终结果：

- 一只最终项目宠物
- 一个项目总评分
- 一张可分享海报
- 一段项目总结

示例结果：

```text
你的项目宠物：赛博灵感猫
当前阶段：Lv.4 成长期
项目评分：82
最强属性：趣味性 91
当前弱点：稳定性 64
宠物评语：这个项目很有传播感，但需要进一步收敛 MVP。
```

这让项目从过程反馈变成可展示、可传播的成果。

## 5. UI 页面设计

### 5.1 页面一：项目蛋首页

目标：建立第一印象，让用户知道这是一个会被输入孵化的项目生命体。

核心元素：

- 居中的发光项目蛋
- 输入框
- 初始引导语

页面文案：

```text
Press Enter. Hatch your project.
Tell me your project idea...
```

交互：

- 用户输入第一个想法并按下回车
- 项目蛋发光
- 蛋体震动
- 裂纹出现
- 进入宠物主界面

视觉氛围：

- 白色或浅灰背景
- 轻赛博蓝光效
- 像素风，但避免廉价感
- 可爱、治愈、轻科技

### 5.2 页面二：宠物主界面

目标：承载主要使用流程，是产品的核心页面。

布局：

```text
┌────────────────────────────────────────┐
│              Project Pet               │
│                                        │
│   左侧评分       中间宠物       右侧日志 │
│                                        │
│           输入框 Press Enter            │
└────────────────────────────────────────┘
```

区域说明：

- 中间：宠物主体，需要最大、最醒目，是产品记忆点。
- 左侧：五维项目属性评分。
- 右侧：Agent 评估日志。
- 底部：输入框，承载持续 Vibe Coding。

左侧评分示例：

```text
清晰度 72
完成度 58
趣味性 81
稳定性 64
传播性 77
```

右侧日志示例：

```text
本轮变化：项目增加了传播机制。
Agent 判断：趣味性提升，但复杂度略升。
下一步建议：先做扫码闭环，不要急着做排行榜。
```

### 5.3 页面三：进化瞬间页或弹窗

目标：制造现场展示的记忆点。

触发条件示例：

- Lv.1 项目蛋 -> Lv.2 初始宠物
- Lv.2 初始宠物 -> Lv.3 成长期宠物
- Lv.3 成长期宠物 -> Lv.4 完全体

动画流程：

```text
屏幕微暗 -> 宠物发光 -> 像素粒子散开 -> 白光闪烁 -> 新形态出现 -> 等级提升
```

提示文案：

```text
Your project has evolved.
你的项目进化了。
```

### 5.4 页面四：负面状态与修复反馈

目标：当项目风险上升时，用可爱但明确的方式提示用户修复。

状态示例：

- 迷茫：眼睛变成问号，头轻微左右晃动。
- 受伤：身体变暗，心形核心闪烁。
- 过载：周围出现 glitch 像素闪烁。

注意：负面状态不能恐怖，仍然要保持可爱和可修复感。

文案示例：

```text
它有点过载了。你的项目功能变多了，但 MVP 变得不清晰。
```

修复建议示例：

```text
试试输入：“帮我压缩成 3 个核心功能。”
```

这一页的重点是让用户产生“我可以救回这个项目宠物”的行动欲望。

### 5.5 页面五：最终海报页

目标：生成可展示、可分享的项目宠物报告卡。

海报内容：

- 项目宠物形象
- 项目名称
- 总评分
- 五维雷达图或进度条
- 进化次数
- 最强属性
- 当前弱点
- 宠物评语

示例：

```text
VibePet Report

我的项目宠物：Cyber Hatch Cat
项目评分：82
最强属性：趣味性
当前弱点：稳定性
进化次数：5
本轮评语：这是一个很有传播潜力的项目，但需要收敛技术范围。
```

最终海报是产品的传播点。

## 6. 动画顶层设计

动画采用状态驱动，不做复杂骨骼动画。每种宠物状态绑定一种简单但明确的 CSS 动效。

### 6.1 Idle Float 待机漂浮

用途：平时待机。

表现：

- 宠物上下轻轻漂浮
- 心形核心微弱闪烁
- 尾巴轻微摆动或两三帧切换

意义：

```text
项目处于正常生长状态。
```

### 6.2 Enter Pulse 输入反馈

用途：用户按下回车。

表现：

- 输入框发光
- 能量流向宠物
- 宠物轻微放大再恢复
- 身上青蓝色光点闪一下

意义：

```text
这次 prompt 被宠物吸收了。
```

### 6.3 Happy Bounce 开心跳动

用途：项目变清晰、变好玩。

表现：

- 宠物跳一下
- 眼睛变亮
- 周围出现小星星或像素粒子
- 评分条上涨

意义：

```text
本次修改是正向迭代。
```

### 6.4 Confused Shake 迷茫晃动

用途：需求不清晰。

表现：

- 宠物头歪一下
- 身体轻轻左右晃动
- 头上出现问号像素
- 清晰度下降或保持不变

意义：

```text
用户需要更明确地描述项目。
```

### 6.5 Injured Flicker 受伤闪烁

用途：稳定性下降。

表现：

- 宠物整体变暗
- 心形核心闪烁
- 出现小裂纹或低电量 icon
- 稳定性条下降

意义：

```text
项目可实现性或技术稳定度存在风险。
```

### 6.6 Overload Glitch 过载故障

用途：功能堆太多。

表现：

- 宠物周围出现 glitch 像素
- 身体短暂错位
- 眼睛变成螺旋或紧张表情
- 出现复杂度提示

意义：

```text
项目变得太重，需要收敛。
```

### 6.7 Evolve Flash 进化闪光

用途：达到关键节点。

表现：

- 宠物被光圈包围
- 像素粒子向外扩散
- 白光一闪
- 切换到新形态
- 出现等级提升

意义：

```text
项目完成一次阶段性成长。
```

### 6.8 Asking Question 主动追问

用途：Agent 触发追问，宠物向用户提问。

表现：

- 宠物停止当前动画，切换到歪头姿态
- 眼睛交替闪烁问号像素
- 气泡弹出追问文字，带闪烁光标效果
- 心形核心缓慢脉冲，颜色从青色变为思考暖黄
- 等待用户回应
- 用户回应后：正面回答→宠物开心跳一下；模糊回应→宠物微歪头追问

意义：

```text
宠物不只是执行者，而是思考伙伴——它在帮你想清楚。
```

## 7. Agent 与 UI 对接协议

前后端通过固定 JSON 解耦。Agent 负责状态判断和追问生成，UI 负责状态视觉化和追问呈现。

示例返回：

```json
{
  "petState": "happy",
  "evolutionStage": 2,
  "score": 78,
  "metrics": {
    "clarity": 82,
    "completion": 61,
    "fun": 88,
    "stability": 66,
    "shareability": 79
  },
  "delta": {
    "clarity": 8,
    "completion": 5,
    "fun": 12,
    "stability": -3,
    "shareability": 7
  },
  "summary": "这个改动让项目更有趣，也更适合现场传播。",
  "risk": "功能数量增加后，MVP 可能变得偏重。",
  "suggestion": "建议先完成扫码互动和宠物反馈闭环。",
  "question": {
    "text": "如果明天就要演示，这三个新增功能里哪一个最能让观众记住你的项目？",
    "trigger": "连续功能堆叠",
    "category": "功能取舍"
  }
}
```

`question` 字段为可选——当本轮不触发追问时为 `null`。

状态映射：

| petState | UI 表现 |
| --- | --- |
| egg | 项目蛋 |
| idle | 待机漂浮 |
| happy | 开心跳动 |
| confused | 迷茫晃动 |
| injured | 受伤闪烁 |
| overload | glitch 过载 |
| evolve | 进化光效 |
| asking | 歪头提问 + 气泡弹出问题 |
| final | 完全体展示 |

解耦原则：

- Agent 只负责状态判断和追问生成。
- UI 只负责状态视觉化和追问呈现。
- 动画只依赖 `petState`、`evolutionStage`、`metrics`、`delta` 和 `question`。

## 8. UI/动画负责人任务清单

## 9. MVP 闭环

最小可展示版本需要完成以下闭环：

```text
用户输入 → 回车 → Agent 返回 JSON（评分 + 状态 + 追问）→ 宠物吸收能量 → 状态变化 → 分数变化 → 日志反馈 → 触发进化或负面修复 → 触发追问或建议
```

MVP 阶段 Agent 可用规则模拟，但需要同时覆盖**反馈**和**追问**两条线：

**反馈规则**：
- 输入包含”简化””MVP””核心功能”：提升清晰度和稳定性。
- 输入包含”分享””海报””邀请”：提升传播性和趣味性。
- 输入包含”排行榜””AI””地图””实时””多人”：提升趣味性，但可能降低稳定性。
- 输入过短或模糊：触发 confused。
- 功能连续增加过多：触发 overload。

**追问规则**（核心差异点）：
- 用户连续 3 轮输入都只加功能不做收敛：触发一次追问（随机从追问库抽取）。
- 稳定性连续下降 2 轮：触发”范围收敛”类追问。
- 用户方向跳跃：触发”定位确认”类追问。
- 追问后用户正面回应（输入包含简化/收敛/明确回答）：宠物进入 happy。
- 追问后用户忽略继续堆功能：本轮不自问自答，但 overload 概率升高。

MVP 的追问库可硬编码 3-5 条精选问题，不需要 AI 生成。只要**宠物开口提问**这个行为本身成立，反空心化的概念就成立。

只要**反馈 + 追问**两条线都跑通，项目概念就成立。

## 10. 答辩表达

完整表达：

> 我们做的是一个会陪你 Vibe Coding 的 AI 项目桌宠。它不只是被动反馈项目状态，还会在用户盲目堆功能时主动追问"这个功能解决的是真实痛点，还是只是看起来很酷？"。我们用三种核心能力——项目状态反馈、项目理解追问、反空心化陪伴——把沉默的代码变成会说话的老师，让养成感从装饰变成思考驱动力。用户在每一次回车中孵化项目，在每一次追问中成为更清醒的创作者。

简短表达：

> Vibe Coding 让人变成点菜员——加登录、加排行榜、加 AI。我们的宠物会在你点太多菜的时候问你一句："这道菜真的必要吗？"它把失控的 Vibe Coding 变成一只会提醒你思考的项目宠物。

UI/动画展示重点：

```text
回车 → 宠物吸收能量 → 状态变化 → 评分反馈 → 宠物主动追问 → 用户思考回应 → 进化或修复
```

三个必须在现场被看见的关键瞬间：

1. **反馈瞬间**：输入一个模糊需求 → 宠物进入 confused（迷茫晃动 + 问号）→ 观众看到"需求不清晰会有后果"。
2. **追问瞬间**：连续堆了 3 个功能 → 宠物停止执行，歪头弹出问题 → 观众看到"宠物不只是工具，而是思考伙伴"。
3. **进化瞬间**：回应追问 + 做出关键收敛 → 宠物触发进化（光圈 + 粒子 + 白光）→ 观众看到"好的思考让项目蜕变"。

只要这三个瞬间在现场被看见，Vibeuddy 的核心价值就能被理解。
